## Architecture

The repo has two top-level parts:

- **Root Swift Package** — the library itself. `Package.swift`, `Sources/`, and `Tests/` live at the repo root, so the package can be consumed directly via an SPM remote URL.
- **`WebParserDemo/`** — an Xcode app (generated by XcodeGen from `project.yml`) demonstrating library usage.

### Library internals (`Sources/WebParser/`)

The parsing flow chains four components in order:

```
WebParser (entry)
  → WebParserSession (Cookie / data store singleton)
  → WebParserEngine (WKWebView off-screen rendering engine)
  → WebParserMapper (data transformer)
```

**`WebParser`** (`@MainActor class`) — orchestrates the full lifecycle: Cookie sync → Engine execution → retry loop → Mapper mapping. Callers provide a `WebParserConfig` and any `WebParserMapper`; the return type is the generic `M.T`.

**`WebParserEngine`** (`@MainActor class`) — wraps `WKWebView` in a `CheckedContinuation<Data, Error>`. After the page loads it enters a polling loop, repeatedly calling `evaluateJavaScript` until the result is non-empty or it times out. The result is serialized to `Data` via `JSONSerialization` before returning — this is the key bridge point for `Sendable` compliance. `config` and an `onStateChange` closure are injected via the initializer, so the Engine has no direct dependency on `WebParser`.

**`WebParserMapper`** (protocol) — handles the `Data → T` transformation, with two built-in implementations:
- `WebParserJSJSONMapper<T: Decodable>` — decodes via `JSONDecoder`. Use when `executionJS` returns a JS object or array.
- `WebParserRegexMapper<T>` — delegates the transformation to a caller-supplied `(Data) throws -> T` closure; the return type is not restricted to `Codable`. Use when `executionJS` returns plain text (e.g. `document.title` or full HTML). Note: despite the name, this mapper performs no regex — it is a generic closure transformer.

**`WebParserSession`** (`@MainActor` singleton) — holds the shared `WKWebsiteDataStore`, ensuring Cookies and cache are correctly shared across parsing tasks.

**`WebParserState`** (enum) — the state machine, dispatched via `WebParserProgressDelegate`: `started → loading(progress:) → executingJavaScript → completed | retrying | failed`.

### Demo app structure (`WebParserDemo/WebParserDemo/`)

The Demo uses a UIKit SceneDelegate architecture following MVVMC. Three groups:

- **`App/`** — `AppDelegate` (`@main`) and `SceneDelegate` (builds a `UITabBarController` + two `UINavigationController`s).
- **`TitleTest/`** — fetches a page title from a URL, using `WebParserRegexMapper`.
- **`ComicList/`** — parses a comic update list, using `WebParserJSJSONMapper` with `WebParserProgressDelegate` to show load progress.

Each feature follows a `HostController → ViewModel → View` three-layer structure. ViewModels use `@Observable`, and each feature keeps its models in a `+Models.swift` file.

### Key design decisions

- All WebKit operations are marked `@MainActor` because `WKWebView` must run on the main thread.
- The Engine returns `Data` (not `Any`) to satisfy `Sendable` across concurrency boundaries.
- JavaScript scripts should use Swift raw strings (`#"""..."""#`) to avoid escaping issues.
- `blockMedia = true` (default) blocks video/audio loading via `mediaTypesRequiringUserActionForPlayback`; keep it on for better performance.

## Code Style

- **Formatter**: run `swiftformat .` before every commit; config is `.swiftformat` at the repo root.
- **Indentation**: 2 spaces.
- **Acronyms**: all caps, e.g. `URL`, `ID`, `JS`, `JSON`, `UUID`.
- **Trailing commas**: always add a trailing comma at the end of collection literals.
- **MARK blocks**: use `MARK: -` to separate sections; order members by visibility.
- **DocC comments**: all public APIs must have Chinese DocC comments, using half-width English punctuation (comma `,`, period `.`).
- **Test framework**: Swift Testing (`@Test`, `#expect`), not XCTest.
- **Swift version**: 6.2, with Strict Concurrency checking enabled.

## Future Plans

### Kotlin Android port

Plan to port WebParser to a native Kotlin Android version, with an architecture that maps 1:1 to the iOS version:

| iOS (Swift) | Android (Kotlin) |
|-------------|-----------------|
| `@MainActor class` | `@MainThread` + `Dispatchers.Main` |
| `async throws` | `suspend` |
| `CheckedContinuation` | `suspendCancellableCoroutine` |
| Swift protocol + associatedtype | Kotlin interface + generics |
| Swift enum (associated values) | Kotlin sealed class |
| `WKWebView` | Android `WebView` |
| `WKNavigationDelegate` | `WebViewClient` |
| KVO `estimatedProgress` | `WebChromeClient.onProgressChanged()` |
| `WKWebsiteDataStore` | `android.webkit.CookieManager` |
| `JSONSerialization → Data` | not needed (`evaluateJavascript` returns a JSON String directly) |
| `CancellationError` | `CancellationException` |

**Differences from iOS (need extra handling):**
1. Android `WebView` must be attached to the view hierarchy to render — create an invisible `FrameLayout` attached to the window.
2. JavaScript must be enabled manually: `webView.settings.javaScriptEnabled = true`.
3. Timeout must be implemented manually via `withTimeout { }`.
4. `evaluateJavascript` returns the string `"null"` for JS `null`, so `isResultPresent` needs extra handling.

The `WebParserMapper` input type changes from `Data` to `String` (the Android version receives the JSON string directly).
